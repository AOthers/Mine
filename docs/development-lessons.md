# 开发踩坑记录

这份文档记录本项目开发过程中已经遇到过的问题。后续继续开发前建议先扫一遍，避免重复犯同样的错。

## 产品与页面设计

### 功能设置不要放错位置

问题：一开始把百度网盘 API 配置放到了「我的」页，和用户预期不一致。

原因：「备份与恢复」是独立工具，百度网盘配置属于这个工具自己的设置，不属于全局个人设置。

以后避免：

- 和某个工具强相关的配置，优先放在该工具的设置页。
- 全局设置或账号资料才放到「我的」页。
- 「备份与恢复」页不要再加右上角齿轮入口，设置入口通过状态卡片的「管理/去配置」进入。

### 应用名和功能名必须分清

问题：页面和文档里曾经混用「应用备份」「我的」「备份与恢复」。

以后避免：

- 软件名固定为「我的」。
- 功能名使用「备份与恢复」「影视」等具体工具名。
- README、页面标题、图标描述、发布说明要区分产品名和功能名。

### 底部导航和系统返回要一起设计

问题：手机侧滑返回曾经不管在哪个页面都直接退到系统桌面。

原因：只做了页面切换，没有把 Android 系统返回和应用内导航栈统一处理。

以后避免：

- 子页面先返回所属功能页。
- 顶层非「功能」tab 先回到「功能」tab。
- 只有位于主功能页时，系统返回才退出应用。
- 新增工具页面时同步检查系统返回逻辑。
- WebView 页面要优先处理网页历史，再退出工具页面。

## 影视 WebView

### Compose 依赖版本不一致会导致页面闪退

问题：点击「影视」后曾经闪退，Logcat 显示：

```text
java.lang.NoSuchMethodError:
No virtual method at(Ljava/lang/Object;I)
androidx.compose.animation.core.KeyframesSpec$KeyframesSpecConfig
```

原因：`CircularProgressIndicator` 触发了当前 Material3 和 Compose animation 版本之间的运行时不兼容。代码能编译，但运行时调用了 APK 里不存在的方法。

以后避免：

- 遇到 Compose 组件闪退时，不要只看业务代码，要先看 Logcat 栈顶的 Compose/Material 库版本。
- 若栈在 Material3 动画组件里，优先用更稳定的简单 UI 替代，或统一升级 Compose BOM、Material3、compiler extension。
- 影视页当前避免使用会触发该问题的 `CircularProgressIndicator`。

### WebView 功能边界要明确

问题：影视功能容易被误解成要解析播放源或抓取影视网站内容。

以后避免：

- v1 只做用户主动浏览：打开网页、站内跳转、刷新、回首页、外部浏览器打开。
- 不解析网页结构。
- 不提取播放地址。
- 不绕过网站限制、付费、版权或广告机制。
- 错误状态只提供重试和浏览器打开。

## 备份与恢复

### 切换网盘备份目录后刷新不能留下旧数据

问题：用户修改百度网盘备份目录后点击刷新，列表里仍显示旧目录里的备份应用。

原因：异步加载远程备份记录时，没有在刷新前清空旧列表，也没有丢弃旧路径返回的晚到结果。

以后避免：

- 上传和远程列表读取都必须使用 `tokenStore.getBackupPath()`。
- 修改网盘路径后立即刷新远程备份记录。
- 刷新远程记录前先清空旧数据。
- 异步加载时记录本次加载使用的路径，如果返回时路径已变化，丢弃旧结果。

### 恢复列表不要只显示包名

问题：恢复应用页面曾经只显示包名，不显示应用名。

原因：远程备份记录如果只依赖包名，换手机后本机可能没有该应用，无法通过包名反查应用名。

以后避免：

- 新备份文件名使用 `<应用名>_<包名>_<版本>.apk`。
- 恢复列表优先从备份文件名解析应用名。
- 兼容旧格式 `<包名>_<版本>.apk`。
- 已下载 APK 管理页优先读取 APK 元数据，显示应用名、包名、版本和图标。

### 应用图标不能只依赖当前手机已安装应用

问题：恢复列表和已下载列表曾经图标缺失或显示成占位图。

原因：恢复场景可能发生在新手机上，本机没有安装旧应用，不能总是从 `PackageManager` 读取已安装应用图标。

以后避免：

- 本机已安装应用可以从 `PackageManager` 获取图标。
- 已下载 APK 要从 APK archive 信息读取应用图标和版本信息。
- 远程云端备份没有本地 APK 时允许显示占位图，但不要把占位图当作真实应用图标。

### 同版本备份要有策略

问题：同一个软件重复上传时曾经会直接覆盖，用户无法确认是否保留旧文件。

以后避免：

- 版本不同：自然保留多个版本。
- 版本相同：按设置页策略处理。
- 默认策略是覆盖。
- 另存策略使用 `_copy-yyyyMMddHHmmss` 追加到文件名，避免冲突。
- 恢复页点击应用后必须列出该应用的所有版本。

## 百度网盘

### 上传路径和恢复列表路径必须一致

问题：曾经出现「检测不到之前备份的软件」。

原因：上传和远程列表读取没有完全使用同一个保存路径，或者切换路径后旧结果仍留在页面上。

以后避免：

- 上传和远程列表都必须使用 `tokenStore.getBackupPath()`。
- 修改网盘路径后刷新恢复列表。
- 刷新远程记录前清空旧数据。
- 如果异步返回的结果来自旧路径，必须丢弃。

### `filemetas` 接口不要用错

问题：网盘恢复曾经提示「获取下载链接失败：filemetas 失败」。

原因：百度网盘接口路径和参数不一致时，错误码不一定直观，容易误判为下载失败。

以后避免：

- 列表接口和文件信息接口分别使用正确的百度网盘 API。
- 获取下载链接前确认 `fs_id`、`path`、`access_token` 都来自同一条远程记录。
- 错误提示要带出具体阶段，例如列表失败、filemetas 失败、下载失败。
- 修百度网盘问题时优先看 `BaiduPanService.kt`，不要只改 UI。

### OAuth 成功后要回到业务页面

问题：授权成功后曾经回到软件首页，而不是「备份与恢复」页。

原因：OAuth 回调只处理了登录状态，没有恢复用户授权前所在的业务上下文。

以后避免：

- OAuth 成功后回到「备份与恢复」页。
- 成功后刷新远程备份记录。
- 以后新增外部授权流程，也要保存并恢复用户业务上下文。

### API 教程要放在设置页

问题：用户需要知道怎么获取百度网盘 AppKey 和 SecretKey。

以后避免：

- 教程入口放在「备份与恢复」设置页右上角。
- 教程内容包含 AppKey、SecretKey、应用创建和 OAuth 回调地址。
- 当前 OAuth 回调地址是 `wode://baidu.oauth`。

## 恢复安装包保存路径

### 不要把 SAF 选择的目录当成普通文件路径

问题：自定义恢复路径一开始只想显示为普通路径，后来发现 Android 文件夹选择器返回的是 `primary:` 或 tree URI，用户看不懂。

原因：Android 存储权限和 SAF 文件夹选择器不是普通字符串路径输入。`primary:Android/data/...` 是系统文档树标识，不等于真实磁盘路径字符串。

以后避免：

- 恢复路径选择使用 Android 系统文件夹选择器。
- 保存的是 SAF tree URI，不一定是真实文件系统路径。
- 下载、列出、删除、安装都要兼容 `DocumentFile` 和 content URI。
- 显示给用户时可以解码成更可读的路径片段，但不要假装它一定是完整真实路径。
- 不要为了迎合显示效果强行给路径补假的 `Android/data` 前缀。

### 路径显示不能只显示最后一级文件夹

问题：恢复安装包保存位置曾经只显示最后的文件夹名，用户看不出到底选了哪里。

以后避免：

- 默认路径显示应用外部目录下的实际保存位置。
- SAF 路径显示应尽量保留从 `Android/...` 或用户选择树开始的可读片段。
- 文案要说明这是「当前保存位置」，不要让用户误以为是完整可手输路径。

### 已下载 APK 管理页必须显示完整信息

问题：已下载的安装包页面曾经看不到版本号。

以后避免：

- 已下载列表至少显示：图标、应用名、包名、版本、大小、时间、路径。
- 文本空间不足时优先换行，不要省略关键版本信息。
- 删除按钮不要挤占版本号区域。

## 通知与安装

### 备份和恢复都要有通知栏进度

问题：长时间上传或下载时，用户无法判断任务是否还在进行。

以后避免：

- 备份上传显示通知进度。
- 网盘恢复下载显示通知进度。
- Android 13 及以上要处理通知权限。
- 失败时通知和页面提示要能说明失败阶段。

### 下载完成后要明确 APK 放在哪里

问题：恢复完成后用户找不到安装包。

以后避免：

- 默认恢复目录是应用外部文件目录下的 `restored_apks`。
- 设置页显示当前保存位置。
- 恢复页右上角提供已下载 APK 管理入口。
- 管理入口支持查看、安装和删除。

## 收藏功能

### 功能卡片长按收藏要支持反向操作

问题：只有收藏入口不够，已经收藏时还需要取消收藏。

以后避免：

- 未收藏时长按弹出「收藏」。
- 已收藏时长按弹出「取消收藏」。
- 收藏页只显示已收藏工具。
- 新增工具卡片时要同步接入收藏状态。

## 检查更新与 GitHub Release

### App 内更新必须依赖 GitHub Release 的 APK 资产

问题：只推送 commit 或 tag 不够，App 检查更新即使发现版本更新，也没有 APK 可下载。

原因：`UpdateService` 读取 GitHub latest release，并选择 release asset 里的第一个 `.apk` 文件。

以后避免：

- 发布新版本时必须创建 GitHub Release。
- Release tag 建议使用 `v1.1` 这种带 `v` 的格式。
- Release 里必须上传 `.apk` 资产，例如 `Mine-v1.1.apk`。
- App 版本号可以是 `1.1`，更新比较会忽略 tag 前面的 `v`。
- 没有 release asset 时，App 无法完成自动下载更新。

### 本机没有 `gh` 或 token 时不能自动创建 Release

问题：开发过程中可以推送 `master` 和 tag，但无法直接创建 GitHub Release 并上传 APK。

原因：本机没有 GitHub CLI `gh`，环境里也没有 `GH_TOKEN` 或 `GITHUB_TOKEN`。

以后避免：

- 若要自动发布，先安装并登录 `gh`，或配置可用的 GitHub token。
- 没有发布凭据时，只能本地生成 APK，再到 GitHub Release 页面手动上传。
- 手动上传前确认 tag 已推送到远端。

### debug 包和 release 包大小差很多是正常现象

问题：v1.1 release APK 只有约 1.86 MB，而 debug APK 约 17.7 MB，看起来像少打了功能。

原因：debug 包包含调试工具、Compose tooling、额外元数据；release 包启用了 R8/ProGuard 压缩混淆，会删掉未使用代码和资源。

以后避免：

- 对外发布使用 release APK。
- 比较包大小时必须区分 debug 和 release。
- release 体积变小不代表功能缺失，优先用安装运行和 `assembleRelease` 结果确认。

## 发布与 GitHub

### release 签名文件不能进仓库

问题：需要生成正式 APK，但签名私钥不能上传 GitHub。

以后避免：

- `mine-release.jks` 只保存在本地。
- `keystore.properties` 只保存在本地。
- `.gitignore` 必须包含：

```gitignore
*.jks
keystore.properties
```

- Gradle 配置要做到：本地有 `keystore.properties` 时启用 release 签名；没有时不影响别人 clone 项目。

### R8 缺失注解类不一定是业务代码问题

问题：`assembleRelease` 曾经因为 `com.google.errorprone.annotations.*` 缺失失败。

原因：`security-crypto` / Tink 依赖里引用了编译期注解，R8 会在 release 混淆时检查到。

以后避免：

- 遇到 R8 missing classes，先看 `app/build/outputs/mapping/release/missing_rules.txt`。
- 如果是编译期注解类，按生成建议加 `-dontwarn`。
- 不要为了绕过错误直接关闭 release 构建验证。

### 移动标签后必须强制推送标签

问题：版本标签如果在本地移动过，远程标签不会自动更新。

以后避免：

```powershell
git tag -f v1.1
git push origin master
git push origin v1.1 --force
```

推送后检查：

```powershell
git log --oneline --decorate -3
git status -sb
git ls-remote --tags origin v1.1
```

## 文档与编码

### README 和 docs 不要留下乱码

问题：README、AGENTS 和开发记录曾经出现中文乱码。

原因：历史内容编码或终端显示不一致，导致文档看起来像无效信息。

以后避免：

- 面向中文用户的文档使用中文，并在提交前打开检查中文是否正常显示。
- README 写给用户和维护者。
- AGENTS.md 只写下次 AI 开发必须遵守的稳定规则。
- 踩坑复盘写到本文档，不要塞进 AGENTS.md。

### AGENTS.md 是规则手册，不是流水账

问题：开发过程中容易把历史过程填进 Agent 说明文件。

以后避免：

- 踩坑复盘写到本文档。
- README 写给开源用户。
- AGENTS.md 只写下次 AI 开发必须遵守的稳定规则。
- 不要把单次开发流水账塞进 AGENTS.md。

## 工具与环境

### Gradle wrapper 可能被沙箱网络拦截

问题：`assembleDebug` 或 `assembleRelease` 曾经卡在下载 Gradle distribution，并报 `Permission denied: getsockopt`。

原因：当前开发环境的网络被沙箱限制，Gradle wrapper 需要访问 `services.gradle.org`。

以后避免：

- 这是环境权限问题，不是项目代码问题。
- 构建失败时先看是否停在 `Downloading https://services.gradle.org/distributions/...`。
- 在受限环境里需要放行 Gradle wrapper，或提前准备好本地 Gradle 缓存。

### 不要把 IDE 自动文件混进功能提交

问题：Android Studio 可能自动修改 `.idea/gradle.xml`、`.idea/misc.xml`，并生成 `.idea/runConfigurations.xml`。

以后避免：

- 功能提交前先看 `git status --short`。
- 如果 `.idea` 变化和本次功能无关，不要一起提交。
- 真正需要提交 IDE 配置时，要单独说明原因。

## 提交前检查清单

每次功能收尾前至少检查：

- 页面入口位置是否符合产品逻辑。
- 系统返回是否符合导航规则。
- 新功能是否需要接入收藏、搜索、设置或通知。
- 备份上传路径和恢复读取路径是否一致。
- 新手机场景下是否还能显示应用名、版本和图标。
- SAF 路径是否按 content URI 处理。
- WebView 页面是否处理加载失败和返回历史。
- 检查更新是否有 GitHub Release 和 `.apk` asset。
- `assembleDebug` 是否通过。
- 涉及发布配置时，`assembleRelease` 是否通过。
- 私钥、密码、APK、build 输出是否被 `.gitignore` 忽略。
- README、AGENTS 和相关 docs 是否需要同步。

## 音乐播放器

### 文件夹扫描必须缓存并由来源变化驱动

问题：SAF 文件夹递归扫描比 `MediaStore` 系统音乐查询慢很多，每次打开音乐页都扫描会让播放器像卡住。

以后避免：
- 选中的音乐文件夹和已识别曲目缓存放在 `MusicStore`。
- App 进程还活着时，重新进入音乐页复用现有曲库。
- 只在重新启动 App、用户手动刷新、或音乐来源变化时重新扫描。
- 移除文件夹来源时同步清理对应文件夹缓存。

### 歌词结果可能包含不可显示的行

问题：联网歌词或解析结果可能出现空行、只有时间标记的行，甚至字面量 `null`。这些内容会导致顶部歌词闪回“暂无歌词”或显示 `null`。

以后避免：
- 在解析或 ViewModel 层过滤空行、`null`、只有括号时间标记的行。
- 当前歌词到下一句之前要保持上一句，不要在时间间隔内切回空状态。
- 顶部歌词区高度固定，长句用横向滚动，不要撑高页面。

### 应用列表权限弹窗只属于备份与恢复入口

问题：配置好百度网盘后弹出查看应用列表权限，会让用户误以为网盘授权触发了不相关权限。

以后避免：
- 不要在 `AppListViewModel` 初始化时加载已安装应用。
- 用户进入备份与恢复时再触发应用列表加载。
- 百度网盘授权和应用列表权限是两个独立意图，不要混在一起。

### 手动检查更新要绕过已跳过版本

问题：启动时尊重“跳过此版本”是对的，但用户手动点击 `检查更新` 表示想重新检查。

以后避免：
- 启动检查可以跳过已忽略的 tag。
- `我的` 页面手动检查更新时忽略 skipped tag。
